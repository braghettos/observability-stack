# Runbook: Widget Resolution Failure

## Trigger
Snowplow error logs matching one of these widget-specific patterns:
- `unable to resolve api reference` → **widget-missing-restaction** (Widget references a non-existent RESTAction via `resourceRefId`)
- `unable to resolve widgetDataTemplate` → **widget-template-error** (jq template in `widgetDataTemplate` failed)
- `unable to resolve widget` → **widget-error-generic** (schema validation failure on the Widget CR)

## Severity
P3 (Portal UI degraded — a page/panel/button renders empty or errors; platform healthy)

## Primary Specialist
**krateo_portal_agent** (widgets are Portal CRs). Delegate RESTAction fixes to
`krateo_restaction_agent` when the root cause is the referenced RESTAction.

## Diagnosis (via krateo_observability_agent)

### Step 1 — Identify the failing widget
```sql
SELECT toString(Timestamp) AS ts,
       JSONExtractString(Body, 'msg') AS msg,
       JSONExtractString(Body, 'widget', 'kind') AS widget_kind,
       JSONExtractString(Body, 'widget', 'name') AS widget_name,
       JSONExtractString(Body, 'widget', 'namespace') AS widget_ns,
       JSONExtractString(Body, 'err') AS error_detail,
       JSONExtractString(Body, 'traceId') AS trace_id
FROM otel_logs
WHERE ResourceAttributes['k8s.pod.name'] LIKE 'snowplow-%'
  AND (Body LIKE '%unable to resolve api reference%'
       OR Body LIKE '%unable to resolve widgetDataTemplate%'
       OR Body LIKE '%unable to resolve widget%')
  AND Timestamp > now() - INTERVAL 30 MINUTE
ORDER BY Timestamp DESC LIMIT 20
```

### Step 2 — Classify
Match the `msg` against the 3 patterns. The `widget.kind` tells you the CRD
type: `Panel`, `Button`, `Form`, `DataGrid`, `EventList`, `Route`, etc.

## Decision Tree

### widget-missing-restaction (`unable to resolve api reference`)
**Cause:** The Widget spec has a `resourceRefId` (or `items[].resourceRefId`,
`actions.rest[].resourceRefId`, `autocomplete[].resourceRefId`) that points at
a RESTAction that does not exist, or exists in a different namespace.

**Diagnose:**
```
k8s_get_resources: restactions.templates.krateo.io in namespace <widget_ns>
```
Compare the returned list against the `resourceRefId` values in the Widget spec.

**Remediate (via krateo_portal_agent):**
- If the RESTAction is missing entirely: create it, OR update the Widget's
  `resourcesRefs` to point at an existing RESTAction with matching shape.
- If the RESTAction exists in a different namespace: RESTActions must live in
  the same namespace as the Widget. Move/recreate it in the widget namespace.

**Hint for operators:** Widgets reference RESTActions via **`resourceRefId`**
(NOT `apiRef`). The reference ID is resolved against the widget's
`resourcesRefs` map populated by Snowplow when rendering the parent page.

### widget-template-error (`unable to resolve widgetDataTemplate`)
**Cause:** The Widget's `spec.widgetDataTemplate` jq expression failed to
execute against the resolved api data map.

**Diagnose:** Extract the jq template from the Widget CR:
```
k8s_get_resources: <widget.kind>.widgets.templates.krateo.io <widget_name> -n <widget_ns>
```
Dump the underlying API response first to understand its shape, then test
the jq locally:
```
echo '<api-response>' | jq -c '<widgetDataTemplate>'
```

**Remediate (via krateo_portal_agent):**
- Fix the jq expression in the Widget CR.
- Common pitfalls: path that doesn't exist, type mismatch (string vs array),
  unsupported builtin (gojq omits some).

### widget-error-generic (`unable to resolve widget`)
**Cause:** JSON-schema validation against the Widget CRD's `openAPIV3Schema`
failed, or the unstructured→typed conversion failed.

**Diagnose:** The log `err` field typically contains the exact validation
error ("required", "additionalProperties not allowed", "must validate one and
only one schema", etc.).

**Remediate (via krateo_portal_agent):**
- Read the Widget CRD's openAPIV3Schema:
  ```
  k8s_get_resources: crd <widget.kind>s.widgets.templates.krateo.io -o yaml
  ```
- Compare the Widget CR spec against the schema. Fix missing required fields
  or remove invalid fields.

## Verification (via krateo_observability_agent)
Wait 60 seconds after fix, then:
```sql
SELECT count() AS error_count
FROM otel_logs
WHERE ResourceAttributes['k8s.pod.name'] LIKE 'snowplow-%'
  AND (Body LIKE '%unable to resolve api reference%'
       OR Body LIKE '%unable to resolve widgetDataTemplate%'
       OR Body LIKE '%unable to resolve widget%')
  AND JSONExtractString(Body, 'widget', 'name') = '<widget_name>'
  AND JSONExtractString(Body, 'widget', 'namespace') = '<widget_ns>'
  AND Timestamp > now() - INTERVAL 2 MINUTE
```
**Success criterion:** `error_count = 0`

## Escalation
- **If the Widget is generated by a Krateo composition** (not user-authored):
  the fix belongs in the CompositionDefinition template that generates it.
  Delegate to `krateo_blueprint_agent` instead of patching the Widget directly
  — otherwise the next reconciliation will overwrite the fix.
- **If the widget schema keeps drifting from the CRD:** the SchemaDefinition
  upstream may need to be updated. See `composition_failure` runbook.
